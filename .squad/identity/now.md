# What We're Focused On

The current repo state still centers on the backend-owned session animation surface in `backend/app/services/animation.py` and `backend/app/api/router.py`, with semantic animation ids resolved from lifecycle state and exposed on `/session/animation` for both snapshot reads and live SSE delivery. That backend contract remains semantic-only: it resolves animation ids and lifecycle state, not raw per-bone channel arrays. The viewer continues to resolve generated humanoid runtime sidecars locally from those semantic ids and consume the preserved `channels` payload directly.

Jason has now also provided a screenshot of the intended final punch frame as the target pose for the current `gesture.punch.once` endpoint work. That reference arrived after the exporter/runtime audit had already narrowed the missing-motion question: Link confirmed that `scripts/animation_tools/unity/RawAnimBatchExporter.cs` is not sampling VRM-retargeted avatar bones and is not reading per-bone FBX hierarchy transforms, but the current path still exports Unity clip bindings plus derived hints and the generated artifacts already retain real limb channels.

Because of that earlier audit, the punch tuning work stayed local to `frontend/src/avatar/runtime/humanoidChannelPlayback.ts` rather than reopening exporter or backend transport seams. In the first reference-driven pass, Switch preserved the landed lower-leg correction and existing finger behavior, reduced shoulder `front_back` carry further, slightly increased hand `down_up` and `in_out`, and strengthened punch-specific arm `front_back` carry to reduce shoulder-led carry and improve wrist and hand alignment at the endpoint. The frontend build passed after that runtime pass.

Live browser validation on the 4175 display surface showed that first pass materially improved the punch endpoint against the supplied reference while preserving stable legs and avoiding the older shoulder flare, but it also left a remaining low guard height. Switch then ran a second narrow pass and concluded that this residual mismatch was punch-specific upper-arm carry, not lower-arm quaternion interpretation. That follow-up changed only the `gesture.punch.once` overrides for `left/right.arm.down_up` and `left/right.arm.front_back`, pushing elbows and forearms farther forward and slightly higher while leaving shoulders, wrists, hands, lower-body bindings, and fingers unchanged. The frontend build passed again after that pass.

The current state is therefore narrower and more concrete than before: `gesture.punch.once` is now materially closer to the provided Unity reference, the stable lower-body and hand baseline remains intact, and the old shoulder-led flare has not returned. The remaining mismatch is no longer framed as gross bone wiring, missing export, lower-leg instability, or lower-arm quaternion delivery. It is now a finer guard-pose calibration problem inside the punch-specific upper-arm carry. Visible backend speech lifecycle 422 responses remain outside the scope of this punch-tuning slice.

Because Jason needs to move PCs, the repo is being checkpointed with a concrete exporter handoff rather than another runtime guess. The next session should extend `scripts/animation_tools/unity/RawAnimBatchExporter.cs` additively so it keeps the current `unity_humanoid_muscle` channels intact while also sampling humanoid bone-local transforms from `Animator.GetBoneTransform(HumanBodyBones)` for a minimal comparison set: upper arms, lower arms, hands, upper legs, lower legs, and feet. Scope the first pass to `gesture.punch.once` only, emit explicit comparison metadata between the current muscle-space channels and the sampled humanoid-bone-space rotations, regenerate that one clip, and compare the final frame against the existing dev-only browser override surface before widening to `idle.default`. If the bone-space result is better, the next step after that experiment is to extend the exporter approach across the shared clips and simplify the runtime's remaining arm-space guesswork. Throughout that follow-up, backend contracts should stay semantic-only, and `.tmp-unity-temp` cache noise should stay out of git commits.
